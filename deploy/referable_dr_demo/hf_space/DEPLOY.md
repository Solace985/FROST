# Deploying FROST to Hugging Face Spaces (Docker) — runbook

A single **Docker Space** serves the whole app (FastAPI backend **and** static
frontend) in one CPU container. This is the correct architecture — Vercel
serverless cannot hold a resident PyTorch model (250 MB function cap, ephemeral
cold starts, per-request reloads), so it is not used for the backend.

## What travels where (the weights question, answered)

| Artifact | Size | Where it comes from | Upload needed? |
| --- | --- | --- | --- |
| RETFound-Green backbone | 84 MB | **Public** GitHub release (Apache-2.0), fetched + SHA-verified in the Dockerfile at build | **No** |
| Trained MultiTaskHead | 501 KB | Travels *inside* the Space (staged by `assemble_space.py`) | No (auto-copied) |
| Operating point (threshold + hashes) | ~1 KB | Travels inside the Space (auto-copied) | No |

**So you do NOT need a separate weights repo.** Recommended route: the head is
tiny and rides along in the Space; the backbone auto-downloads at build.
(An optional HF *model-repo* route exists — see "Alternative" at the end — if you
prefer to keep the trained head in a **private** repo.)

---

## Step 0 — one-time prerequisites (on your machine)
- A Hugging Face account + the `huggingface_hub` CLI: `pip install -U huggingface_hub` then `hf auth login` (or `huggingface-cli login`).
- `git` (and `git-lfs` if you like, though the head is small enough not to need it).
- The local artifacts already generated by the demonstrator:
  - the trained head `runs/train/brset_20260525_113425/model_checkpoint.pt`, and
  - `deploy/referable_dr_demo/.local/operating_point.local.json` (from `analysis/derive_threshold.py`).

## Step 1 — run the LOCAL parity gate (mandatory; do not deploy unless it passes)
This proves the code reproduces the study scores on real BRSET images **before**
you ship. It runs only on your machine (BRSET is credentialed and never leaves it).

```bash
uv run python deploy/referable_dr_demo/analysis/verify_parity.py
```
**PASS looks like** (exit code 0):
```
SUMMARY
  canonical synthetic parity : pass (max abs logit diff=…e-…, max abs score diff=…e-…)
  study-linked parity        : pass (n=5, max abs diff=…e-…)
```
If `study-linked parity` is `unavailable`, point it at your local BRSET test
images/predictions and rerun until it says **pass**. Do not proceed on `fail`.

## Step 2 — create the Space
1. https://huggingface.co/new-space → **Owner** = you, **Space name** = e.g. `frost-referable-dr`.
2. **Space SDK = Docker** (blank template). **Hardware = CPU basic (free)**.
3. Visibility: **Public** (shareable URL) or **Private** (only you). See the note
   on the head in "Considerations" below.
4. Create. Note the git URL, e.g. `https://huggingface.co/spaces/<you>/frost-referable-dr`.

## Step 3 — stage the Space contents (one command)
Clone the empty Space and assemble it from the repo:
```bash
git clone https://huggingface.co/spaces/<you>/frost-referable-dr /tmp/frost-space

uv run python deploy/referable_dr_demo/hf_space/assemble_space.py --space-dir /tmp/frost-space
```
This copies (no credentialed data, no backbone): `Dockerfile`, `.dockerignore`,
`README.md` (Space card), `src/retina_screen/`, the 3 config YAMLs,
`deploy/referable_dr_demo/` (minus tests/.local/caches), the **head** →
`hf_space/weights/mt_head_native392.pt`, and the **operating point** →
`hf_space/operating_point.json`. It prints a size summary and the git commands.

## Step 4 — push
```bash
cd /tmp/frost-space
git add -A
git commit -m "FROST referable-DR demonstrator (Docker Space)"
git push
```
(No environment variables to set for the recommended route — the Dockerfile bakes
them in. `FROST_ALLOW_SYNTHETIC_ONLY_READINESS=1` is set in the image.)

## Step 5 — first boot (what to expect)
On the Space's **Logs** tab you'll see, in order:
1. **Build** (a few minutes): pip installs CPU torch/deps; the backbone is fetched
   and the line `retfoundgreen_statedict.pth: OK` confirms the SHA-256 check.
2. **Run**: `INFO frost.app FROST ready (bundle=frost-referable-dr-1).`
   - If instead you see `FROST not ready: …`, read the reason (missing head,
     hash mismatch, or threshold/bundle problem) and fix that artifact.
3. The app is at `https://<you>-frost-referable-dr.hf.space` (HTTPS, automatic).

**Confirm it works:** open the URL, upload a JPEG/PNG fundus image, tick the
confirmation box, click **Run prediction** → you get a REFERABLE / NOT REFERABLE
decision, the threshold gauge, and the full pipeline trace. `GET /health` returns
`{"status":"ready", "parity_status":"pass", …}`.

## Step 6 — free-tier behaviour (so you're not surprised)
- The Space **sleeps after ~48 h of inactivity**; the next visit **wakes it in a
  few seconds** (a fresh model load, ~1–3 s on CPU). No data is lost.
- CPU inference for one image is typically ~0.3–1 s after warm-up.
- Restarts/rebuilds re-fetch the backbone from GitHub (cached in the image layer,
  so only a rebuild re-downloads).

## Cross-cutting confirmations
- **CORS:** the frontend and `/predict` are served by the **same** FastAPI app on
  the **same origin** → no CORS configuration is needed (and none is added).
- **HTTPS:** provided automatically by Spaces; no certificate work.
- **Memory:** resident model ≈ RETFound-Green ViT-S (~22M params, ~88 MB) + a
  ~0.5 MB head + CPU-torch runtime. Comfortably within the free tier's 16 GB RAM.
  The model is a **process-wide singleton loaded once at startup** (see
  `backend/service/inference.py::init_service`), never per request.
- **No storage:** uploads are processed in memory and discarded per request; no
  database, accounts, or image persistence.
- **Scope/disclaimer framing:** intact and shown in the running UI (the Space card
  banner, the in-app footer's two not-a-medical-device statements, and the About
  panel). This matters more on a public URL, not less.
- **License:** RETFound-Green weights are **Apache-2.0** (redistribution/hosting
  permitted; attribution preserved via the Space card + config provenance). The
  trained head is the authors' own and is freely hostable.

## Considerations before you make it public
- Making the Space **public** publishes the study code (`src/retina_screen`) and
  the 501 KB **trained head**. If your paper is unpublished and you'd rather not
  expose the trained head yet, use the **Alternative** below (private model repo)
  or make the **Space private** (URL only works while logged into your account).

## Alternative — keep the head in a PRIVATE Hugging Face model repo
If you prefer not to commit the head into the Space:
1. Create a **model** repo (not a Space), e.g. `https://huggingface.co/new` →
   Model → `frost-weights` → **Private**.
2. Upload two files to it, named exactly:
   - `mt_head_native392.pt`  ← your `runs/.../model_checkpoint.pt` (the head)
   - `retfoundgreen_statedict.pth`  ← the RETFound-Green backbone (optional; if
     omitted, the Dockerfile's public fetch is still used)
   ```bash
   hf upload <you>/frost-weights runs/train/brset_20260525_113425/model_checkpoint.pt mt_head_native392.pt
   ```
3. In the Space → **Settings → Variables and secrets**:
   - Variable `FROST_WEIGHTS_REPO = <you>/frost-weights`
   - Secret `HF_TOKEN = <a read token for that private repo>`
4. Redeploy. `hf_space/entry.py` will download the head (and backbone if present)
   from the model repo at startup and the bundle gate re-verifies hashes/shapes.
   Do **not** commit the head into the Space in this route (remove it from
   `hf_space/weights/` before pushing, or it will simply be overridden).
